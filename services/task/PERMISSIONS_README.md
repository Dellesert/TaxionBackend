# Система прав доступа к задачам

## Обзор

Реализована гибкая система прав доступа для задач с поддержкой делегирования, подзадач и чеклистов.

## Структура прав доступа

### TaskPermissions

```go
type TaskPermissions struct {
    CanView              bool // Просмотр задачи
    CanViewSubtasks      bool // Просмотр всех подзадач
    CanEdit              bool // Редактирование задачи
    CanChangeStatus      bool // Изменение статуса
    CanCheckItems        bool // Отмечание чеклистов
    CanCreateSubtasks    bool // Создание подзадач
    CanDelegate          bool // Делегирование задачи
    CanEmergencyComplete bool // Аварийное завершение
    CanAssignUsers       bool // Назначение исполнителей
    CanDelete            bool // Удаление задачи
}
```

## Роли пользователей

### 1. Creator (Создатель задачи)
**Полный доступ ко всем функциям созданной задачи (основной или подзадачи):**
- ✅ Просмотр задачи и всех подзадач
- ✅ Редактирование задачи
- ✅ Изменение статуса задачи
- ✅ Отметка чеклистов
- ✅ Создание подзадач
- ✅ Делегирование
- ✅ Аварийное завершение
- ✅ Назначение пользователей
- ✅ Удаление задачи

**Важно:** Роль Creator применяется к любой задаче, которую вы создали (основной или подзадаче), независимо от того, назначена она вам или другому пользователю. Роль Creator имеет приоритет над ролью Assignee.

### 2. Current Assignee (Текущий исполнитель основной задачи)
**Права на выполнение задачи:**
- ✅ Просмотр задачи и всех подзадач
- ❌ Редактирование (не может менять название, описание и другие поля)
- ✅ Изменение статуса
- ✅ Отметка чеклистов
- ✅ Создание подзадач
- ✅ Делегирование
- ❌ Аварийное завершение
- ❌ Назначение пользователей (только создатель может назначать)
- ❌ Удаление задачи

### 3. Subtask Assignee (Исполнитель подзадачи)
**Ограниченные права только на свою подзадачу:**
- ✅ Просмотр своей подзадачи
- ❌ Просмотр других подзадач
- ❌ Редактирование описания
- ✅ Изменение статуса (только своей подзадачи)
- ✅ Отметка чеклистов (только в своей подзадаче)
- ❌ Создание подзадач
- ❌ Делегирование
- ❌ Аварийное завершение
- ❌ Назначение пользователей
- ❌ Удаление

### 4. Parent Task Creator (Создатель родительской задачи, НЕ создавший данную подзадачу)
**Права на контроль и управление подзадачами, созданными другими:**
- ✅ Просмотр подзадачи
- ✅ Просмотр всех других подзадач
- ❌ Редактирование подзадачи
- ❌ Изменение статуса подзадачи
- ❌ Отметка чеклистов в подзадаче
- ❌ Создание под-подзадач
- ❌ Делегирование подзадачи
- ⚠️ Аварийное завершение (только если подзадача просрочена >3 дней)
- ❌ Назначение пользователей
- ✅ Удаление подзадачи

**Важно:** Эта роль применяется только когда вы создали родительскую задачу, но **подзадачу создал кто-то другой** (например, текущий исполнитель родительской задачи). Если вы сами создали подзадачу, то получаете роль Creator с полными правами.

### 5. Delegation Chain (Бывший исполнитель)
**Права на просмотр и аварийное завершение:**
- ✅ Просмотр задачи и всех подзадач
- ❌ Редактирование
- ❌ Изменение статуса
- ❌ Отметка чеклистов
- ❌ Создание подзадач
- ❌ Делегирование
- ⚠️ Аварийное завершение (только если задача просрочена >3 дней)
- ❌ Назначение пользователей
- ❌ Удаление

## API Эндпоинты

### 1. Получение прав доступа к задаче
```http
GET /api/v1/tasks/{id}/permissions
```

**Ответ:**
```json
{
  "permissions": {
    "can_view": true,
    "can_view_subtasks": true,
    "can_edit": true,
    "can_change_status": true,
    "can_check_items": true,
    "can_create_subtasks": true,
    "can_delegate": true,
    "can_emergency_complete": false,
    "can_assign_users": true,
    "can_delete": false
  },
  "request_id": "..."
}
```

### 2. Получение задачи с правами
```http
GET /api/v1/tasks/{id}
```

**Ответ включает поле `permissions`:**
```json
{
  "task": {
    "id": 123,
    "title": "Task title",
    ...
    "permissions": {
      "can_view": true,
      "can_edit": true,
      ...
    }
  },
  "request_id": "..."
}
```

### 3. Аварийное завершение задачи
```http
POST /api/v1/tasks/{id}/emergency-complete
```

**Условия:**
- Пользователь должен быть в цепочке делегирования
- Задача должна быть просрочена более чем на 3 дня

**Ответ при успехе:**
```json
{
  "message": "Task completed (emergency)",
  "request_id": "..."
}
```

**Ответ при отсутствии прав:**
```json
{
  "error": "Emergency completion is only allowed for overdue tasks (>3 days) by users in delegation chain",
  "request_id": "..."
}
```

## Использование в коде

### Проверка прав в usecase

```go
import "tachyon-messenger/services/task/permissions"

// Получение прав пользователя
perms, err := permissions.GetTaskPermissions(ctx, task, userID)
if err != nil {
    return err
}

// Проверка конкретного права
if !perms.CanEdit {
    return fmt.Errorf("access denied: cannot edit task")
}
```

### Использование middleware для проверки прав

```go
import taskmiddleware "tachyon-messenger/services/task/middleware"

// В setupRoutes:
permMiddleware := taskmiddleware.NewPermissionsMiddleware(taskUsecase)

// Применение к маршруту
protected.PUT("/tasks/:id",
    permMiddleware.RequirePermission("edit"),
    taskHandler.UpdateTask)

protected.PATCH("/tasks/:id/status",
    permMiddleware.RequirePermission("change_status"),
    taskHandler.UpdateTaskStatus)
```

## Логика определения роли

Система автоматически определяет роль пользователя в задаче в следующем порядке:

1. **Subtask Assignee** - если пользователь в списке assignees подзадачи
2. **Current Assignee** - если пользователь в списке assignees основной задачи
3. **Parent Task Creator** - если пользователь создатель подзадачи, но НЕ её исполнитель (т.е. создал подзадачу для другого)
4. **Creator** - если пользователь создатель основной задачи
5. **Delegation Chain** - если пользователь в `DelegatedFromUserID` или `OriginalAssigneeID`
6. **None** - нет доступа

### Важно:
- Проверка исполнителя имеет приоритет над проверкой создателя
- Это позволяет правильно различать создателя родительской задачи от исполнителя подзадачи

## Особые случаи

### Отметка чеклистов

Пользователь может отмечать чеклисты только если:
1. Имеет право `can_check_items`
2. Является исполнителем **этой конкретной** задачи (не подзадачи другого пользователя)

### Аварийное завершение

Доступно только для пользователей в цепочке делегирования, когда задача просрочена более чем на 3 дня.

При аварийном завершении:
- Задача переводится в статус `done`
- Устанавливается `completed_at`
- Логируется activity с флагом `emergency: true`

## Миграция и обратная совместимость

Система полностью обратно совместима с существующим кодом. Поле `permissions` опционально и добавляется автоматически при запросе задачи через `GET /api/v1/tasks/{id}`.

## Тестирование

Для тестирования системы прав:

1. Создайте задачу как пользователь A
2. Делегируйте её пользователю B
3. Проверьте права пользователя A (должен быть в delegation chain)
4. Создайте подзадачу для пользователя C
5. Проверьте права пользователя C (должен видеть только свою подзадачу)
6. Измените deadline на прошедшую дату (-4 дня)
7. Проверьте `can_emergency_complete` для пользователя A (должно быть true)
